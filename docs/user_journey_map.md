## 一、為什麼要做使用者旅程

傳統需求分析常從「系統功能」出發,容易產生「功能很多、但沒人用」的窘況。
本文件改採以**使用者為中心**(User-Centered Design)的思考方式,先描繪真實使用情境,再推導出系統應提供的功能。

此方法亦能確保:
- 文件章節之間有共同錨點
- 每個功能皆有真實使用情境背書
- 系統設計具備可擴充性

## 二、目標使用者識別(Persona)

依組織導入 ISMS 之常見角色分布,本系統服務對象區分為以下 3 個 Persona。
角色定義參考 ISO 27001:2022 Clause 5.3 中對 ISMS 角色與責任之規範。

### Persona 1:小陳(評估者 / Assessor)

| 屬性 | 內容 |
|---|---|
| 職位 | 資訊部一般工程師(兼 ISMS 執行) |
| 年資 | 1-3 年 |
| 對 ISMS 的熟悉度 | 中低(知道有 ISO 27001,但不熟控制條文) |
| 主要任務 | 維護資產清冊、執行年度風險評鑑 |
| 工作頻率 | 每月維護、每年全面評鑑一次 |
| 痛點 | 1. CIA 不知道怎麼填<br>2. 算出分數後不知道怎麼處理<br>3. 主管要報告但格式不一致 |
| 對系統的期待 | 「告訴我哪些是高風險、我該做什麼就好」 |

### Persona 2:王經理(管理員 / Administrator)

| 屬性 | 內容 |
|---|---|
| 職位 | 資訊部主管 / 資安長 |
| 年資 | 8+ 年 |
| 對 ISMS 的熟悉度 | 高(熟悉 ISO 27001、稽核流程) |
| 主要任務 | 審核評鑑結果、向高層報告、應付稽核 |
| 工作頻率 | 季度檢視、稽核前密集審核 |
| 痛點 | 1. 員工填的 CIA 標準不一致<br>2. 稽核時找不到歷史紀錄<br>3. 不知道誰改過什麼 |
| 對系統的期待 | 「員工填的東西要可信、可追溯、可呈現給稽核員」 |

### Persona 3:陳稽核員(稽核者 / Auditor)— 間接使用者

| 屬性 | 內容 |
|---|---|
| 職位 | 外部 ISMS 稽核員(BSI / SGS 等) |
| 角色 | 不直接使用系統,但會檢視系統輸出 |
| 關注重點 | 評鑑過程可追溯、CIA 評估有依據、控制措施有對應 |
| 痛點 | 受評組織常無法說明「為什麼這個資產 CIA 是 9」 |
| 對系統的期待 | 「給我看評鑑邏輯,證明不是隨便填的」 |

## 三、使用者旅程(7 階段)

以 Persona 1「小陳」為主角,描繪完整工作週期。

### 📍 階段 1:首次接觸 —「這系統能幫我什麼?」

**情境**:小陳剛被指派負責 ISMS,主管 LINE 告知「下週稽核要看資產風險評鑑表,請今天前更新完。」

**使用者狀態**:
- 心情:焦慮(時間緊迫)
- 認知:模糊(不確定 ISMS 怎麼做)
- 行為:Google 搜尋「ISMS 工具」、「資產評鑑系統」

**接觸點(Touchpoint)**:
- 進入 RiskGenie 首頁

**使用者期待**:
- 30 秒內知道「這系統做什麼」
- 知道「我能不能用」
- 看到「從哪裡開始」

**痛點 / 障礙**:
- 看到專業術語(CIA、RAG、ISO 27002)但不知道意思
- 不確定要不要註冊
- 介面複雜不知道按哪個

**設計需求**:
- 提供一頁簡介(系統做什麼)
- 提供「3 分鐘上手」教學連結
- 介面術語應有 hover 提示

---

### 📍 階段 2:註冊登入 —「我能用嗎?」

**情境**:小陳決定試用,但他公司還沒幫他開帳號。

**使用者狀態**:
- 心情:猶豫(我可以註冊嗎?)
- 認知:不熟悉系統權限機制
- 行為:點「登入」→ 沒帳號 → 找「註冊」

**接觸點**:
- 登入頁、註冊頁、Email 驗證信

**使用者期待**:
- 用公司 email 註冊
- 註冊後立刻可以用(或至少知道何時可用)
- 忘記密碼能自助重設

**痛點 / 障礙**:
- 不知道註冊後是不是自動核准
- 不知道密碼規則
- 不知道帳號要不要主管核准

**設計需求**:
- UC1 登入登出 含註冊功能
- 新註冊者狀態為「待審核」,由管理員核准
- 密碼安全:至少 8 位含英數
- 註冊需 Email 驗證

---

### 📍 階段 3:建立資產清冊 —「我的 Excel 怎麼辦?」

**情境**:小陳登入後,要把公司既有的資產清單放進系統。手上有 Excel,但欄位跟系統不同。

**使用者狀態**:
- 心情:挫折(欄位對不到)
- 認知:不知道怎麼處理「不標準」的資料
- 行為:嘗試上傳 → 失敗 → 想放棄

**接觸點**:
- 資產上傳介面、欄位對應介面、手動新增介面

**使用者期待**:
- 上傳 Excel,系統能識別欄位(就算欄位名不同)
- 上傳失敗有明確錯誤訊息
- 也能手動一筆一筆新增

**痛點 / 障礙**:
- Excel 欄位是「資產編號」、系統要 asset_id
- 部分資產沒填 CIA,被系統擋下
- 35 筆資料一筆一筆手動輸入太慢

**設計需求**:
- UC2 資產新增:支援 Excel 批次上傳 + 手動新增
- AI 欄位辨識(進階):LLM 自動辨識 Excel 欄位對應
- 範本下載
- CIA 缺值時可由 AI 補建議
- 上傳失敗有明確訊息與修復指引

---

### 📍 階段 4:執行風險評鑑 —「AI 怎麼幫我?」

**情境**:30 筆資料上傳完成,要為每筆評估 CIA。小陳不知道「客戶資料庫機密性是 7 還是 9」。

**使用者狀態**:
- 心情:焦慮(怕填錯被稽核員質問)
- 認知:對 ISMS 評分邏輯不熟
- 行為:看到「AI 評估」按鈕,猶豫要不要點

**接觸點**:
- 風險評鑑頁面、AI 建議對照表、權重調整介面

**使用者期待**:
- AI 給的建議要有理由(不要黑箱)
- 可以修改 AI 給的值(我才是負責人)
- 可以比對人工值與 AI 建議的差異
- 可以保留歷史(下次重評可對照)

**痛點 / 障礙**:
- AI 給了一個值,但不知道怎麼來的
- 想調整 CIA 權重(金融業可用性比機密性重要),不知道能不能
- 不確定 AI 建議是否能對抗稽核員質問

**設計需求**:
- UC7 風險評鑑:AI 對每筆資產建議 CIA + 理由
- 三選一機制:採用 AI / 修改 AI / 維持人工值
- 差異標記:cia_delta ≥ 5 自動標記建議複核
- 權重調整:CIA 權重可由使用者調整
- UC8 風險值計算:規則引擎計算(可解釋)
- UC9 風險分級:分為 Low/Medium/High/Critical

---

### 📍 階段 5:查看結果 —「Critical 怎麼處理?」

**情境**:評鑑完成,系統顯示「5 筆 Critical 風險」。小陳第一次看到 Critical,不知道嚴重程度,也不知道下一步該做什麼。

**使用者狀態**:
- 心情:焦慮 + 困惑(這 5 個怎麼辦?)
- 認知:不熟 ISO 27002 控制條文
- 行為:點開 Critical 資產 → 想看「具體建議」

**接觸點**:
- 風險儀表板、單筆資產詳情、AI 建議卡片、ISO 條文連結

**使用者期待**:
- 看到白話的風險說明(不是只看到分數)
- 拿到具體的改善動作(不是抽象建議)
- 知道對應到哪條 ISO 規範
- 標記「已處理 / 待處理」狀態

**痛點 / 障礙**:
- 系統只給「分數 92, Critical」,但不知道意思
- 給建議但太抽象(「加強存取控制」是要加強什麼?)
- 不知道做完該怎麼記錄

**設計需求**:
- UC10 AI 改善建議:三段式輸出(風險摘要 + 改善動作 + 預期效益)
- 每個建議標註 ISO 27002 條文
- 提供可執行的 SOP 草案(進階)
- 標記「待處理 / 處理中 / 已完成」
- 建議文字白話、避免術語堆疊

---

### 📍 階段 6:匯出報告 —「我能交差嗎?」

**情境**:小陳處理完,主管要求「明天會議要看完整報告」。

**使用者狀態**:
- 心情:有把握(系統有評鑑紀錄)
- 認知:主管要的格式不確定
- 行為:點「匯出」→ 選格式

**接觸點**:
- 匯出介面、檔案下載、Email 傳送(進階)

**使用者期待**:
- 多種格式選擇(PDF 給主管、Excel 給稽核、JSON 給其他系統)
- 報告結構清楚
- 含評鑑日期、執行人、依據等 metadata

**痛點 / 障礙**:
- 系統只給 Excel,主管要 PDF
- PDF 沒有公司 logo
- 報告中找不到執行人資訊

**設計需求**:
- UC12 報表匯出:支援 PDF / Excel / JSON 三種格式
- 含日期、執行人、評鑑版本
- PDF 含基本 ISMS 報告結構
- 匯出檔案不可缺漏資產

---

### 📍 階段 7:回訪追蹤 —「上次的紀錄呢?」

**情境**:3 個月後,小陳要做季度檢視,想看「上次評鑑結果有沒有改善」。

**使用者狀態**:
- 心情:期待(看到改善的成就感)
- 認知:已熟悉系統操作
- 行為:點「歷史評鑑」→ 比對

**接觸點**:
- 歷史評鑑列表、差異比對介面、趨勢圖

**使用者期待**:
- 看到歷次評鑑列表(時間、執行人、結果)
- 點兩次評鑑可比對差異
- 看到趨勢圖(Critical 是不是越來越少?)

**痛點 / 障礙**:
- 系統只記錄最新一次,舊資料被覆蓋
- 想看「過去 3 個月新增的資產」找不到
- 想交「改善成果報告」沒辦法產

**設計需求**:
- UC11 風險報表查詢:支援歷史評鑑查詢
- 兩次評鑑差異視覺化
- 風險等級分布隨時間變化(趨勢圖)
- 評鑑紀錄不可被覆蓋(僅新增)
- 所有異動記錄於 AuditLog

## 四、Persona 2「王經理」的差異化旅程

管理員王經理的旅程與小陳有重疊但有差異:

| 階段 | 王經理特有的需求 |
|---|---|
| 註冊登入 | 需要審核新員工帳號 |
| 執行評鑑 | 不會自己做,但要審核小陳的評鑑結果 |
| 查看結果 | 要看全公司風險總覽 |
| 匯出報告 | 要產給董事會看的高層報告 |
| 回訪追蹤 | 要看每個員工的工作量 |

→ 推導出額外需求:
- UC13 帳號權限管理(管理員專屬)
- 待審核帳號審查
- 跨人員彙整報表
- 高層摘要報告

## 五、Persona 3「稽核員」的關注點

陳稽核員雖不直接使用系統,但會檢視系統輸出。她的關注點推導出非功能需求:

- AuditLog 必須記錄所有 CRUD 操作
- 風險分數計算公式可被導出
- 所有紀錄使用統一時區
- 已完成的評鑑紀錄不可修改

## 六、Journey Map 推導出的設計原則

從上述旅程歸納出 5 個系統設計原則:

### 原則 1:人在迴路中(Human-in-the-loop)
AI 給建議,但最終決策權在使用者。系統設計三選一機制(採用 / 修改 / 維持),不強制覆蓋人工值。

### 原則 2:可解釋優先(Explainability First)
所有 AI 輸出必須附理由;所有風險分數必須能重算;所有控制建議必須有 ISO 條文依據。

### 原則 3:容錯為先(Forgiving by Default)
使用者上傳格式不一致 → 系統嘗試辨識而非拒絕;
CIA 未填 → 系統建議而非阻擋;
操作失誤 → 提供 undo / 歷史回溯。

### 原則 4:歷史可追溯(Auditable)
任何操作皆記錄;任何評鑑皆保存版本;
ISMS 不是一次性作業,系統需支援長期演進。

### 原則 5:權限對應 ISMS 角色
至少區分「評估者」與「管理員」,對應 ISMS 實際運作中的角色分工。

## 七、Journey Map 對應到 14 個 Use Case

| Journey 階段 | 主要 Use Case | 補充功能 |
|---|---|---|
| 1. 首次接觸 | (無直接 UC,屬導覽) | 系統首頁 / 教學引導 |
| 2. 註冊登入 | UC1 登入登出 | + 註冊、Email 驗證 |
| 3. 建立資產 | UC2 新增、UC3 修改、UC4 刪除、UC5 查詢 | + AI 欄位辨識、AuditLog |
| 4. 執行評鑑 | UC7 風險評鑑、UC8 計算、UC9 分級 | + 權重調整、差異標記 |
| 5. 查看結果 | UC10 AI 建議 | + 狀態管理 |
| 6. 匯出報告 | UC11 報表查詢、UC12 匯出 | + 多格式 |
| 7. 回訪追蹤 | UC11 報表查詢(歷史模式) | + 趨勢分析 |
| 管理層 | UC13 帳號權限管理 | (Admin 專屬) |
| 系統維運 | UC16 資料備份、UC18 弱點管理 | (背景作業) |
