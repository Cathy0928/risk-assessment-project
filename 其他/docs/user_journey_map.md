## 一、為什麼要做使用者旅程

傳統需求分析常從「系統功能」出發,容易產生「功能很多、但沒人用」的窘況。
本文件改採以**使用者為中心**(User-Centered Design)的思考方式,
先描繪真實使用情境,再推導出系統應提供的功能。

此方法確保:
- 文件章節之間有共同錨點
- 每個功能皆有真實使用情境背書
- 系統設計具備可擴充性

## 二、目標使用者識別(Persona)

依組織導入 ISMS 之常見角色分布,本系統服務對象區分為 3 個 Persona。
角色定義參考 ISO 27001:2022 Clause 5.3 中對 ISMS 角色與責任之規範。

### Persona 1:小陳(評估者 / 一般使用者)

| 屬性 | 內容 |
|---|---|
| 職位 | 資訊部一般工程師(兼 ISMS 執行) |
| 年資 | 1-3 年 |
| 對 ISMS 的熟悉度 | 中低 |
| 主要任務 | 維護資產清冊、執行年度風險評鑑 |
| 工作頻率 | 每月維護、每年全面評鑑 |
| 痛點 | CIA 不知道怎麼填、不知道分數後怎麼處理 |
| 對系統的期待 | 「告訴我哪些是高風險、我該做什麼就好」 |

### Persona 2:王經理(資安管理員 / 高階主管)

| 屬性 | 內容 |
|---|---|
| 職位 | 資訊部主管 / 資安長 |
| 年資 | 8+ 年 |
| 對 ISMS 的熟悉度 | 高 |
| 主要任務 | 審核評鑑結果、向高層報告、應付稽核 |
| 工作頻率 | 季度檢視、稽核前密集審核 |
| 痛點 | 員工填的 CIA 標準不一致、稽核時找不到歷史紀錄 |
| 對系統的期待 | 「員工填的東西要可信、可追溯」 |

### Persona 3:陳稽核員(間接使用者)

| 屬性 | 內容 |
|---|---|
| 職位 | 外部 ISMS 稽核員 |
| 角色 | 不直接使用系統,但會檢視系統輸出 |
| 關注重點 | 評鑑過程可追溯、CIA 評估有依據、控制措施有對應 |
| 痛點 | 受評組織常無法說明「為什麼這個資產 CIA 是 9」 |
| 對系統的期待 | 「給我看評鑑邏輯,證明不是隨便填的」 |

## 三、使用者旅程(7 階段)

以 Persona 1「小陳」為主角,描繪完整工作週期。

### 階段 1:首次接觸 —「這系統能幫我什麼?」

**情境**:小陳剛被指派負責 ISMS,主管 LINE 告知「下週稽核要看資產風險評鑑表,請今天前更新完。」

**使用者狀態**:
- 心情:焦慮(時間緊迫)
- 認知:模糊(不確定 ISMS 怎麼做)
- 行為:Google 搜尋「ISMS 工具」、「資產評鑑系統」

**痛點**:
- 看到專業術語但不知道意思
- 不確定要不要註冊
- 介面複雜不知道按哪個

**設計需求**:
- 提供一頁簡介(系統做什麼)
- 提供「3 分鐘上手」教學連結
- 介面術語應有 hover 提示

---

### 階段 2:註冊登入 —「我能用嗎?」

**情境**:小陳決定試用,但他公司還沒幫他開帳號。

**使用者狀態**:
- 心情:猶豫
- 認知:不熟悉系統權限機制
- 行為:點「登入」→ 沒帳號 → 找「註冊」

**痛點**:
- 不知道註冊後是不是自動核准
- 不知道密碼規則
- 不知道帳號要不要主管核准

**設計需求**:
- 含註冊功能
- 新註冊者狀態為「待審核」,由管理員核准
- 密碼安全:至少 8 位含英數
- 註冊需 Email 驗證

---

### 階段 3:建立資產清冊 —「我的 Excel 怎麼辦?」

**情境**:小陳要把公司既有資產清單放進系統。手上有 Excel,但欄位跟系統不同。

**使用者狀態**:
- 心情:挫折(欄位對不到)
- 認知:不知道怎麼處理「不標準」的資料
- 行為:嘗試上傳 → 失敗 → 想放棄

**痛點**:
- Excel 欄位是「資產編號」、系統要 asset_id
- 部分資產沒填 CIA,被系統擋下
- 35 筆資料一筆一筆手動輸入太慢

**設計需求**:
- 支援 Excel 批次上傳 + 手動新增
- 系統嘗試自動辨識欄位(進階)
- 提供範本下載
- CIA 缺值時可由 AI 補建議
- 上傳失敗有明確訊息與修復指引

---

### 階段 4:執行風險評鑑 —「AI 怎麼幫我?」

**情境**:30 筆資料上傳完成,要為每筆評估 CIA。小陳不知道「客戶資料庫機密性是 7 還是 9」。

**使用者狀態**:
- 心情:焦慮(怕填錯被稽核員質問)
- 認知:對 ISMS 評分邏輯不熟
- 行為:看到「AI 評估」按鈕,猶豫要不要點

**痛點**:
- AI 給了一個值,但不知道怎麼來的
- 想調整 CIA 權重,不知道能不能
- 不確定 AI 建議是否能對抗稽核員質問

**設計需求**:
- AI 對每筆資產建議 CIA + 理由
- 三選一機制:採用 AI / 修改 AI / 維持人工值
- cia_delta ≥ 5 自動標記建議複核
- CIA 權重可由使用者調整
- 規則引擎計算分數(可解釋)
- 分為 Low / Medium / High / Critical 四級

---

### 階段 5:查看結果 —「Critical 怎麼處理?」

**情境**:評鑑完成,系統顯示「5 筆 Critical 風險」。小陳不知道嚴重程度,也不知道下一步該做什麼。

**使用者狀態**:
- 心情:焦慮 + 困惑
- 認知:不熟 ISO 27002 控制條文
- 行為:點開 Critical 資產 → 想看「具體建議」

**痛點**:
- 系統只給「分數 92, Critical」,但不知道意思
- 給建議但太抽象
- 不知道做完該怎麼記錄

**設計需求**:
- AI 改善建議:三段式輸出(風險摘要 + 改善動作 + 預期效益)
- 每個建議標註 ISO 27002 條文
- 提供可執行的 SOP 草案(進階)
- 標記「待處理 / 處理中 / 已完成」
- 建議文字白話、避免術語堆疊

---

### 階段 6:匯出報告 —「我能交差嗎?」

**情境**:小陳處理完,主管要求「明天會議要看完整報告」。

**使用者狀態**:
- 心情:有把握
- 認知:主管要的格式不確定
- 行為:點「匯出」→ 選格式

**痛點**:
- 系統只給 Excel,主管要 PDF
- PDF 沒有公司 logo
- 報告中找不到執行人資訊

**設計需求**:
- 支援 PDF / Excel / JSON 三種格式
- 含日期、執行人、評鑑版本
- PDF 含基本 ISMS 報告結構
- 匯出檔案不可缺漏資產

---

### 階段 7:回訪追蹤 —「上次的紀錄呢?」

**情境**:3 個月後,小陳要做季度檢視,想看「上次評鑑結果有沒有改善」。

**使用者狀態**:
- 心情:期待
- 認知:已熟悉系統操作
- 行為:點「歷史評鑑」→ 比對

**痛點**:
- 系統只記錄最新一次,舊資料被覆蓋
- 想看「過去 3 個月新增的資產」找不到
- 想交「改善成果報告」沒辦法產

**設計需求**:
- 支援歷史評鑑查詢
- 兩次評鑑差異視覺化
- 風險等級分布隨時間變化(趨勢圖)
- 評鑑紀錄不可被覆蓋(僅新增)
- 所有異動記錄於 AuditLog

## 四、Persona 2「王經理」的差異化旅程

管理員王經理的旅程與小陳有重疊但有差異:

| 階段 | 王經理特有需求 |
|---|---|
| 註冊登入 | 需要審核新員工帳號 |
| 執行評鑑 | 不會自己做,但要審核小陳的評鑑結果 |
| 查看結果 | 要看全公司風險總覽 |
| 匯出報告 | 要產給董事會看的高層報告 |
| 回訪追蹤 | 要看每個員工的工作量 |

→ 推導出額外需求:
- 帳號權限管理(管理員專屬)
- 待審核帳號審查
- 跨人員彙整報表
- 高層摘要報告

---

## 五、Persona 3「稽核員」的關注點

陳稽核員雖不直接使用系統,但會檢視系統輸出。
她的關注點推導出非功能需求:

- AuditLog 必須記錄所有 CRUD 操作
- 風險分數計算公式可被導出
- 所有紀錄使用統一時區
- 已完成的評鑑紀錄不可修改

## 六、Journey Map 推導出的設計原則

從上述旅程歸納出 5 個系統設計原則:

### 原則 1:人在迴路中(Human-in-the-loop)
AI 給建議,但最終決策權在使用者。
系統設計三選一機制(採用 / 修改 / 維持),不強制覆蓋人工值。

### 原則 2:可解釋優先
所有 AI 輸出必須附理由;
所有風險分數必須能重算;
所有控制建議必須有 ISO 條文依據。

### 原則 3:容錯為先
使用者上傳格式不一致 → 系統嘗試辨識而非拒絕;
CIA 未填 → 系統建議而非阻擋;
操作失誤 → 提供 undo / 歷史回溯。

### 原則 4:歷史可追溯
任何操作皆記錄;任何評鑑皆保存版本;
ISMS 不是一次性作業,系統需支援長期演進。

### 原則 5:權限對應 ISMS 角色
至少區分「一般使用者」與「管理員」,
對應 ISMS 實際運作中的角色分工。

| 1. 首次接觸 | (無直接作業,屬導覽) |
|
